最近我替 Hermes 做了幾個本地端客製化。看起來像是幾個零散的小修正，但其實都圍繞同一個目標：**讓流程更接近真實工作、讓更新不要把本地修改洗掉、讓卡片狀態能反映實際進度**。

<!--more-->

![Hermes 客製化工作流](/assets/post-imgs/hermes-customization-cover.svg)

## 一、先把 Kanban 的狀態規則收緊

我先修的是看板流程本身。

過去最讓人不安心的地方，是某些程式開發或修改類任務會被太快結案，甚至直接進 `Done`。這會讓看板看起來很漂亮，但實際上任務可能還沒真正驗證完。

所以我把規則調整成：

- **程式開發、修改類任務一定要先進 `blocked`**
- 不能還沒經過 review / 驗證就直接改成 `Done`
- 如果需要人類確認，應該明確停在 blocked，而不是假裝結束

這個調整的核心不是「拖慢進度」，而是讓卡片狀態真的有意義。`Done` 應該代表事情做完了，而不是「worker 暫時沒有下一步了」。

## 二、PR 可以發，但不能自動默默合併

第二個重點是 PR 流程。

我希望 Hermes 在處理卡片時，可以照正常流程：

1. 先完成程式修改
2. 發出 PR
3. 讓人類 review
4. 再決定要不要合併

所以我把「自動把 PR 合併到 `main`」這種行為排掉了。

這件事很重要，因為它保留了最後一道人工確認：

- 避免未經 review 的變更直接進主線
- 避免 worker 自己替整個專案做 commit 的最終決策
- 讓 PR 真正成為交接點，而不是自動通關按鈕

## 三、`hermes update` 後要自動回灌本地備份分支

這次最實用的一段，是把「更新後不要遺失本地修改」這件事流程化。

我的做法是：

- 本地修改先 **commit**
- 再 push 到 **備份 repo / 備份分支**
- 執行 `hermes update` 更新到最新上游
- 更新完成後，把備份分支再合回更新後的本地分支

這樣就算 `hermes update` 會碰到檔案變動，也不會把我們自己做的客製化直接洗掉。

簡單講，就是把流程拆成這樣：

```text
本地修改 → 備份分支 → hermes update → 回灌合併 → 驗證 → 推送
```

這次我也真的把流程跑了一次，最後有：

- 成功完成 update
- 成功把備份分支回灌回來
- 成功解掉 merge 衝突
- 成功通過相關 Kanban 測試

## 四、為什麼這樣做比較穩

以前最怕的是兩件事：

- 更新完，自己改過的東西不見了
- 看板狀態太樂觀，導致任務早早結束但實際還沒完成

這次的調整就是在對付這兩個問題。

### 對更新流程

把本地客製化先送到備份分支，再在更新後回灌，等於把「備份」變成正式流程的一部分，而不是臨時救援方案。

### 對看板流程

強制 `blocked`、禁止直接 `Done`、禁止自動 merge，讓看板比較像一個真實的交付機制，而不是一個看起來很順但其實少了確認步驟的流程。

## 五、這次實際改了哪些東西

如果把這次的本地修改濃縮成幾句話，大概是這樣：

- 修正 Kanban 的 respawn / guard 邏輯，避免 `active_pr` 這類條件過度干擾
- 強化 Kanban 流程，讓程式開發與修改任務必須先停在 blocked
- 把 `hermes update` 後的 backup reconciliation 納入工作流程
- 重啟 Hermes gateway，讓這些變更確實載入
- 驗證相關測試通過，確認流程不是只有看起來合理

## 六、這對我日常使用的好處

這種調整最直接的感受是：**我比較不用擔心 Hermes 幫我「做完一半」就把事情收掉**。

現在的流程比較像：

- 做事前先交代清楚
- 做到一半就老實 blocked
- 真的完成後才收尾
- 更新時也不怕本地修改遺失

對我來說，這就是 Hermes 真正能長期一起工作的關鍵。

## 結語

我很喜歡把工具慢慢調成自己用得順的樣子。Hermes 也一樣：不是追求把每一步自動化到極致，而是把 **該保留人工確認的地方保留下來**，把 **容易因更新遺失的本地修改保護好**。

這次做完後，我覺得 Hermes 的工作流終於比較像一個能長期維護的系統了。

如果你想看，我下一篇可以直接整理成更實戰版：**「我在 Hermes Kanban 上怎麼寫卡、怎麼拆卡、怎麼讓 worker 不亂收尾」**。
